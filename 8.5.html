HANDLING EXCEPTIONS
problem #   8.5
  1. public class Test {
  2. public static void main(String []args) {
  3. try {
  4. int i = 10/0;
  5. System.out.println("try block statement");
  6. } catch(ArithmeticException ae) {
  7. System.out.println("catch block statement");
  8. return;
  9. } finally {
  10. System.out.println("finally block statement");
  11. }
  12. System.out.println("outside statement");
  13. }
  14. }

Details ...

An exception is thrown (due to an attempt to divide by zero, at line 4) and caught by the catch block (at line 6) – which causes any code inside of the catch to execute before the code flow is then routed to the finally block.

Because the finally block always executes, any code inside of the finally, (i.e. the statement at line 10), like the code inside of the catch block, also runs. After the finally block executes, (in this case) the flow of the code moves to the return statement , at line 8, where its routed back to the calling method and then terminates.

The result of this is that, the code statement at line 12, does not execute.

Interestingly, inserting a return statement inside of the finally block, would make it impossible for the code at line 12 to ever run – resulting in a compiler error (i.e. unreachable statement).

In other words, as currently written (even though the return statement at line 8 routes the code flow back to the calling method) it is possible to include code that subsequently routes to, and executes the statement at line 12 – after the code returns to the calling method. For this reason, no compiler error occurs in this example, even though the statement at line 12 never executes (i.e. it IS reachable).
