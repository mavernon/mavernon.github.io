Details ...

Because method m1 is a static method that is called from within the same class – in which its declared, it can be called without needing to instantiate an object to do so.

Therefore, after the call to m1, and the subsequent throw clause at line 3, the RuntimeException (thrown at line 3) is caught at line 8, and the code flow enters the next try block (at line 9) – where another throw clause occurs.

The catch block for the Exception thats thrown at line 10, contains code logic that both prints a String value, and throws another RuntimeException (at line 13). Since RuntimeException is not required to be caught after its thrown, the execution of the program code is not interrupted as a result of the thrown RuntimeException. Instead, the code flow simply transitions to the next line of program code.

Note: Although the the execution of the program code is not interrupted as a result of the thrown RuntimeException, when the program completes its execution, the program results indicate that a (un-caught) RuntimeException did in fact occur, when the program was run.

The finally block that immediately follows the thrown RuntimeException (at line 13) also contains code logic that prints a String value – and is subsequently followed by another finally block, that prints a String value as well.
